## Verdict

Completed P7X backend interpretation: conditionally accepted. The current
evidence supports using LPS with `orthogonal_drop_adaptive_tiny` and PS-LPS
with `monomial_tiny_ridge` for routine comparisons on this frozen 14-case P7X
suite, provided the claim is framed as a practical default from a descriptive
backend comparison, not as a final universal backend theorem.

S3R-light design: blocked for decision use. The current manifest does not
consistently pair full and screened runs on the same response noise and CV
folds. Half of the intended full/screened pairs have different `response_seed`
and `fold_seed`, so `screened - full` Truth RMSE deltas are confounded with
different noisy responses and fold assignments.

S3R-expanded launch: blocked until the S3R-light manifest and merge/report
contract are fixed. The expanded design is otherwise directionally sound, but
the paired-design contract must be made explicit and machine-checked before
running a larger experiment.

## Blocking Findings

### P1. Auto-chart full/screened pairs use different response and fold seeds

Evidence:

- In `/Users/pgajer/current_projects/geosmooth/scripts/prepare_ps_lps_s3r_light_run.R`,
  lines 80-95 build `task.seed`, `fold_seed`, and `response_seed`.
- The `search.policy` term changes the seed for `screened` tasks in the
  `chart_dim_rule = "auto"` branch.
- Manifest check found 42 matched seed pairs and 42 mismatched seed pairs out
  of 84 intended `(dataset, repetition, chart_dim_rule)` pair keys.

Example from the current manifest:

```text
LA-D1-RAW-N500, repetition 1, auto, full:
  fold_seed = 20361608, response_seed = 20361609

LA-D1-RAW-N500, repetition 1, auto, screened:
  fold_seed = 20361618, response_seed = 20361619
```

The corresponding `local.auto` pair happens to match because of the way the R
`if ... else` expression is parsed, but relying on this precedence accident is
not acceptable.

Required fix:

- Generate a pair-level seed that depends only on dataset, repetition, and
  chart-dimension rule, not on `search_policy`.
- Assign identical `response_seed` and `fold_seed` to `full` and `screened`
  within each intended pair.
- Add `pair_id`, `pair_response_seed`, and `pair_fold_seed` to the manifest.
- Add a manifest validation gate that fails if any intended full/screened pair
  has mismatched response or fold seeds.
- Do not use the current S3R-light auto-pair deltas as support-search policy
  evidence. Treat the current run as smoke/profiling only, or rerun from a
  corrected manifest.

### P1. Pair construction silently drops incomplete or failed pairs

Evidence:

- In `/Users/pgajer/current_projects/geosmooth/scripts/merge_ps_lps_s3r_light_run.R`,
  lines 127-131 only retain pairs where both full and screened rows are `ok`.
- Failed, timed-out, missing, or nonfinite arms are excluded rather than
  represented as pair-level outcomes.

This creates survivorship bias. A screened policy is not acceptable merely
because completed pairs look similar if failures/timeouts are excluded from the
paired table.

Required fix:

- Build the pair table from the full task manifest, then left-join task status
  and result summaries.
- Include both arm statuses in the pair table:
  `full_status`, `screened_status`, `pair_status`, `pair_complete`,
  `pair_exclusion_reason`.
- Report planned pair counts, complete pair counts, incomplete pairs, full-only
  failures, screened-only failures, both-failed pairs, nonfinite pairs, and
  timeouts by dataset and chart-dimension rule.
- Compute accuracy deltas only for valid complete pairs, but keep all planned
  pairs visible in failure/runtime summaries.

### P2. Screened candidate inclusion diagnostics are missing

The current summaries include selected support sizes and candidate counts, but
they do not answer the core mechanism question: did screened search evaluate
the support selected by full search?

Required fix:

- Add `screened_evaluated_supports` to the pair table.
- Add `full_support_in_screened_evaluated_supports`.
- Add `support_match`, `lambda_match`, and, where identifiable,
  `full_selected_candidate_in_screened_candidates`.
- Include these fields in the HTML report and in a linked CSV.

This is not strictly required to compute paired RMSE regret, but it is required
to interpret whether screened search is safely approximating full search or is
winning/losing by choosing different local candidate regions.

### P2. Merge/report status accounting is incomplete for in-progress or missing tasks

The current S3R-light run directory has 168 manifest rows, 114 status JSON files
at audit time, and 104 result RDS files. However the already-written
`tables/task_status.csv` contained only one row because it was generated early.
More generally, the merge script reads only status files that exist and does
not create manifest-backed `missing` or `not_started` rows.

Required fix:

- The merge script should always emit one `task_status.csv` row per manifest
  task.
- Missing status files should be represented explicitly as `missing` or
  `not_started`, not omitted.
- The HTML report should show planned/running/ok/nonfinite/error/missing counts
  by dataset, chart rule, and policy.

## Completed P7X Backend Interpretation

The backend-comparison conclusions are reasonable with the right wording:

- LPS: `orthogonal_drop_adaptive_tiny` is the only tested LPS backend with
  complete coverage across both `auto` and `local.auto` chart rules. The
  monomial and weighted-QR LPS variants have many nonfinite fits, so they are
  not suitable routine LPS defaults yet.
- PS-LPS: `monomial_tiny_ridge` has complete coverage for both `auto` and
  `local.auto`, the smallest median regret in the summary table, and much
  better runtime behavior than the guarded/drop alternatives.
- `weighted_qr_drop_tiny` should not be used in routine broad comparisons. It
  has severe runtime tails in PS-LPS and poor/nonfinite coverage in LPS.

Important caution:

- Some individual datasets are best under other PS-LPS variants, but those
  variants have failures, killed rows, and longer runtime tails. The decision
  should therefore be stated as a robust routine-default choice, not as
  pointwise domination on every dataset.

The backend HTML report is now adequate for the Frank/Friedman-style logic:
metric definitions, regret summaries, runtime, failure accounting, and method
definitions are present. The remaining limitation is the underlying run, not
the report presentation.

## S3R-Light Design Assessment

The intended design is statistically sensible:

- Frozen geometry/truth assets isolate the support-search policy from geometry
  generation variability.
- Repeated response noise and fold assignments are appropriate for estimating
  practical support-search regret.
- Within-pair `screened - full` Truth RMSE is the right primary response, but
  only when both arms share the same geometry, truth, response noise, and CV
  fold assignment.
- Runtime ratio `screened / full` is the right practical response.

Current implementation status:

- The current run is still in progress at audit time.
- A direct status check found 168 manifest rows, 114 status JSON files, 104
  result RDS files, 104 `ok` statuses, and 10 running statuses.
- Because of the seed mismatch, the current auto-chart pairs are not valid
  paired evidence. The local-auto pairs are less affected by the seed issue,
  but a clean rerun is preferable because the manifest-generation expression is
  brittle and the report builder lacks pair-accounting safeguards.

Truth leakage / chart leakage / test-data reuse:

- I did not see evidence that the truth vector is passed into `fit.lps`.
  The worker creates `y` from `asset$f`, fits on `X`, `y`, and `foldid`, and
  computes Truth RMSE after fitting.
- `chart.dim = "auto"` and `chart.dim = "local.auto"` use geometry-derived
  information, which is appropriate for these methods.
- The main leakage-like issue is not truth leakage; it is the paired-design
  seed mismatch.

## S3R-Expanded Recommendations

Do not launch S3R-expanded until the P1 fixes above are in place and validated
on a small regenerated S3R-light manifest.

Recommended repetitions:

- Run a corrected S3R-light with `n_reps = 3` first as a design validation and
  timing smoke test.
- For S3R-expanded, use at least `n_reps = 10`.
- Use `n_reps = 20` if the corrected S3R-light timing shows the full-search
  reference path is feasible under the chosen timeout budget.

Recommended timeout policy:

- Use a predeclared hard timeout for every task; do not rely on manual kills.
- A common 7200-second per-task timeout is acceptable for the corrected
  S3R-light pilot.
- For S3R-expanded, prefer one common timeout across full and screened arms so
  the runtime/failure comparison remains interpretable. If dataset-specific
  thresholds are used, they must be declared before launch and applied
  symmetrically to full and screened within each dataset/chart rule.
- Timeout rows should remain in the pair table and count against the policy
  that timed out.

Recommended report contents:

- Purpose and paired design contract.
- Manifest validation section showing identical seeds within each pair.
- Planned/ok/nonfinite/error/missing/timeout counts by dataset, chart rule, and
  search policy.
- Pair-level full vs screened Truth RMSE scatter with diagonal.
- Distribution of `Delta R = R(screened) - R(full)` by chart rule and geometry
  family.
- Bayesian paired mean/median summaries or a clearly labeled deterministic
  paired summary with intervals.
- Runtime ratio `screened/full`, including tail cases and timeouts.
- Candidate inclusion diagnostics:
  `full_support_in_screened_evaluated_supports`, support match, lambda match,
  screened candidate count, full candidate count.
- Limitations and a recommendation section that separates accuracy regret,
  runtime savings, failure rate, and mechanism evidence.

## Engineering Recommendations

- Keep the current S3R scripts as experiment scripts for now, but avoid creating
  another near-duplicate script set for S3R-expanded. Generalize the existing
  scripts with arguments for `run_id`, `n_reps`, worker count, timeout, and
  screened-search controls.
- Add a manifest QA command to the prepare script and print a short validation
  summary:
  total tasks, total pairs, seed-matched pairs, pair factor balance, expected
  support-grid/candidate settings, and hash coverage for frozen assets.
- Add a merge QA gate that warns or fails when pair seeds differ, when pair
  arms are missing, or when task-status rows are fewer than manifest rows.
- Keep result RDS files untracked unless a separate artifact-storage policy is
  chosen.
