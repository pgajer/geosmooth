## Findings

### [P1] Study (a) verdict can pass without enforcing the SE guard on optimism delta

File: `validation/e1_10_nested_grouped_cv.R`, lines 34--38 and 248--270.

The script predeclares that "the SE of each gated mean" must be below
`0.10/3`, and Study (a)'s gated rule is:

`mean(rel_nested) < 0.10 AND mean(optimism_delta) >= 0`.

The code computes both `se.rel.nested` and `se.delta`, and writes both to the
verdict row, but `se.ok.a` only checks `se.rel.nested`:

```r
se.ok.a <- is.finite(se.rel.nested) && se.rel.nested < SE.MAX
```

This means an acceptance run could report `PASS` even when the optimism-sign
mean is too noisy to support the decision. This is a harness-readiness blocker,
not a criticism of the already-delivered Part A machinery.

Required fix: include `se.delta` in `se.ok.a`, and keep the verdict
`INCONCLUSIVE` unless both gated means satisfy the SE guard.

### [P2] Orchestrator-ratification items remain open before full acceptance

Files:

- `audit_contracts/lps_tiers1to4/e1_10_spec_questions_implementer_2026-06-11.md`, lines 6--8, 10--31, 62--80.
- `project_briefs/lps_tiers1to4_contract_2026-06-11.md`, lines 19--25 and 40--44.

The implementation adds two exported package functions,
`lps.grouped.foldid()` and `lps.nested.cv()`, even though the frozen E1.10
contract names a validation utility and no `fit.lps` signature change. The
implementer correctly pre-submitted this as an additive API proposal, and the
change appears technically reasonable and non-invasive, but the proposal still
needs orchestrator no-objection or amendment before treating the public API as
settled.

The same applies to the Study (b) primary statistic (`nested` under both
foldings) and `K_test = 100`. Both are reasonable predeclarations, but they
are marked as proposals in the spec memo and should be ratified before the
full acceptance run.

### [P3] Integer arguments are silently truncated

File: `R/lps_cv_utils.R`, lines 39--43 and 180--183.

`v <- as.integer(v)` and `inner.folds <- as.integer(inner.folds)` silently
turn values such as `2.9` into `2`. This does not invalidate the E1.10 gates,
but because these are exported utilities, the implementation should reject
non-whole or non-scalar fold counts explicitly.

## Mutation qualification

The auditor temporarily introduced two source mutations in `R/lps_cv_utils.R`,
ran the focused test file, and reverted the source after each probe.

1. Leakage mutation: each inner fit incorrectly received all rows and the
   outer `foldid` instead of `X[train.idx, ]`, `y[train.idx]`, and
   `fold.inner`. Result: focused tests turned red with 18 failures, including
   changed inner CV tables, changed selections, changed predictions, and
   foldid round-trip failures.
2. Cluster-splitting mutation: `lps.grouped.foldid()` returned
   `rep_len(seq_len(v), length(cluster.id))`, ignoring cluster labels.
   Result: focused tests turned red with 13 failures, including whole-cluster,
   fold/train disjointness, leave-cluster-out, and grouped inner-fold failures.

After both mutation probes, the original source was restored. `git diff -- R/lps_cv_utils.R`
was empty, and the focused tests passed again.
