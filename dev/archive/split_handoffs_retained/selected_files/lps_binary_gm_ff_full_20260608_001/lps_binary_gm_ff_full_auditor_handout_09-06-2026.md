## Experiment Being Audited

This experiment compares two binary-outcome LPS modes on synthetic probability
surfaces:

1. `lps_bernoulli_brier`: Bernoulli/Brier LPS, treating binary responses as
   numeric conditional-mean targets and selecting by observed CV Brier score.
2. `lps_binomial_logistic`: binomial/logistic LPS, fitting local logistic
   polynomial models and selecting by observed CV log loss.

The synthetic truth is the probability surface

\\[
  p_i = \Pr(Y_i = 1 \mid X_i).
\\]

The primary goodness-of-fit target is probability Truth RMSE:

\\[
  R(\\hat p,p)
  =
  \\sqrt{
    \\frac{1}{n}
    \\sum_{i=1}^n (\\hat p_i - p_i)^2
  }.
\\]

The main paired comparison is

\\[
  \\Delta_s
  =
  R_s(\\hat p^{\\rm logistic}, p)
  -
  R_s(\\hat p^{\\rm Brier}, p),
\\]

where `s` indexes a matched scenario, repetition, and chart rule. Negative
values favor binomial/logistic LPS; positive values favor Bernoulli/Brier LPS.

## Critical Data-Validity Items To Audit

The first report pass under-scoped the audit as a report-rendering/faithfulness
check. Please explicitly audit the following data-validity issues before
accepting any model interpretation.

### Logistic Fallback Telemetry

The cached full-run CSV has legacy fallback columns:

- `logistic_cv_fallback_event_rate`
- `logistic_final_fallback_event_rate`

Both columns are `NA` for all 11,520 result rows. This is not evidence that
there were no fallback events. It is a run-worker telemetry bug: the worker
recorded obsolete field names, while `fit.lps()` logistic diagnostics use names
such as:

- `event.rate.fallback.fraction`
- `event.rate.fallback.count`
- `fallback.path.fraction`
- `fallback.path.count`
- `na.failure.fraction`
- `na.failure.count`

The worker has been patched for future runs to emit both count and fraction
telemetry. This cached run cannot support claims about logistic local-fit
fallback frequency unless there are separate persisted fit objects or logs that
recover the missing diagnostics.

Audit question: does the HTML report make this limitation unavoidable before
interpreting logistic stability or local-logistic failure behavior?

### Selection-Metric Asymmetry

The two deployed policies were selected by different CV scores:

- Bernoulli/Brier LPS selected candidates by observed CV Brier score.
- Binomial/logistic LPS selected candidates by observed CV log loss.

The primary synthetic target in the report is probability Truth RMSE, which is
closer in spirit to squared probability error than to log loss. Therefore the
current run compares two deployed selection policies, not a clean family
comparison under a common selection metric.

Audit question: does the report avoid saying that local logistic fitting itself
is worse than Bernoulli/Brier fitting without qualifying the selection-score
asymmetry?

### Geometry Reversal

The pooled result hides a large geometry interaction. In the cached paired
table, the logistic method wins every matched pair for the
`1d_highdim_pad100` geometry, while Bernoulli/Brier strongly dominates the 3D
native and high-dimensional geometries.

Audit question: does the report state this geometry dependence clearly enough,
instead of allowing the pooled median to imply one universal winner?

### Scenario-Clustered Uncertainty

The 5,760 paired rows are nested within 288 scenario cells. Treating all paired
rows as independent pseudo-replicates gives overly narrow uncertainty summaries.
The report should either use scenario-clustered summaries for broad claims or
state that pair-level Bayesian-bootstrap intervals are descriptive only.

Audit question: does the report include and interpret scenario-clustered
summaries for broad conclusions?

### Observed Log Loss Scope

The worker computes

```r
observed_logloss = logloss.score(pred, y)
```

after `pred <- fit$fitted.values`, i.e. on the final full-data fitted values.
This is an in-sample diagnostic, not held-out log loss. It must not be used as
a CV or validation diagnostic. A future run should persist fold-held-out
log-loss diagnostics if we want to compare log-loss validation behavior.

Audit question: does the report remove or clearly quarantine observed log loss
as full-data/in-sample rather than held-out?

## Specific Audit Questions

Please answer these explicitly.

1. **Pairing correctness.** Does the report correctly compare
   `lps_binomial_logistic` against `lps_bernoulli_brier` only within matched
   pairs sharing geometry, probability surface, response realization, folds,
   repetition, and chart-dimension rule?

2. **Metric correctness.** Are Truth RMSE, observed log loss, and paired delta
   defined correctly? Are the signs and interpretations of
   `logistic - Bernoulli/Brier` deltas correct throughout?

3. **Selection-score clarity.** Does the report clearly distinguish the
   model-selection scores from the synthetic target? In particular, does it
   avoid implying that observed log loss is the same as probability Truth RMSE?
   Does it make clear that the current policies use asymmetric selection
   metrics?

4. **Fit-status accounting.** Is the all-ok status correctly reported and
   supported by the status rows? Are there any hidden nonfinite, timeout, or
   error paths not visible in the report?

5. **Figure quality.** Do the six figures satisfy the HTML figure/table QC
   checklist? Flag any figure whose labels, legends, captions, or colors need
   revision.

6. **Statistical summary appropriateness.** Are the Bayesian-bootstrap credible
   intervals used appropriately as descriptive paired uncertainty summaries?
   Does the report avoid overstating them as a formal proof of method
   superiority? Are broad claims supported by scenario-clustered summaries,
   not only nested pair-level intervals?

7. **Conclusion support.** Is the headline conclusion supported: in this run,
   the current Bernoulli/Brier selection policy had lower pooled median
   probability Truth RMSE and much faster median runtime than the current
   binomial/logistic selection policy, while geometry-specific reversals remain
   important?

8. **Scope control.** Does the report make clear that this is a controlled
   synthetic probability-surface experiment, not yet a real-data binary-outcome
   validation?

9. **Reproducibility.** Are all linked artifacts present, current, and
   sufficient for another agent to reproduce the report from cached results?

10. **Implementation concerns.** Does the renderer contain any fragile code,
    hard-coded assumptions, stale paths, or ordering assumptions that should be
    fixed before using this pattern for future reports?

11. **Telemetry validity.** Does the report explicitly block logistic fallback
    and stability claims because the cached fallback telemetry is all `NA`?

12. **Future-run readiness.** Is the patched worker sufficient for future runs
    to record logistic fallback counts/fractions and observed log-loss scope?
