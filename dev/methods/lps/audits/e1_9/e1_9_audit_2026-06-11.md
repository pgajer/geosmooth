## Notes And Non-Blocking Issues

The literal audit-assignment command
`Rscript -e 'library(testthat); test_file("tests/testthat/test-lps-bandwidth-multiplier.R")'`
is not self-contained in this worktree: it fails because it does not load the
package, so `fit.lps` and `.klp.kernel.weights` are not found. The committed
bundle harness and the implementer handoff use `pkgload::load_all(".", quiet = TRUE)`
before running the tests; with that package-load step, the focused E1.9 test
file passes with 88 expectations and no failures. This is a procedural command
wording issue, not an implementation blocker for the delivered gates, because
the clean execution bundle explicitly checks pass/fail and exits nonzero on
test failures.

The E1.9 feature is implemented only in the R backend for `b != 1`; this is
consistent with the spec memo and explicit backend guards. The C++ backends
remain unchanged for `b = 1`.

The accepted `b = 0` case remains degenerate and inherited from the existing
zero-weight fallback semantics. It is outside the frozen E1.9 GATE thresholds
and should be revisited only if the orchestrator changes the API contract from
nonnegative to strictly positive multipliers.
